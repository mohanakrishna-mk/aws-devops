Here’s a **clear, exam-focused breakdown of Amazon SQS** based on your notes 👇

---

## 📬 What is SQS?

Amazon SQS (Simple Queue Service) is a:

* **Serverless message queue**
* Fully managed
* Used to **decouple applications**
* Enables **asynchronous processing**

📌 Key idea:
👉 Producer sends message → Queue → Consumer processes later

---

## 🔑 Core Characteristics

* Fully managed (no servers)
* Integrated with IAM
* **Highly scalable (especially Standard queues)**
* No provisioning required 

---

## 📦 Message Size Limit

* Max message size = **1 MB (1024 KB)** 

👉 If message is larger:

1. Upload data to S3
2. Send **S3 object key** via SQS
3. Consumer fetches from S3

---

## 🎯 Why Use SQS?

### 1. Decoupling

* Services don’t depend on each other
* Scale independently

### 2. Async Processing

* No need to wait for response

### 3. Buffering

* Smooths traffic spikes

---

## 🏗️ Common Architectures

### 🧩 1. EC2 / Auto Scaling Consumers

* EC2 instances read from SQS
* Auto-scale based on queue size

---

### ⚡ 2. Lambda + SQS

* Event Source Mapping pulls messages
* Batch size: **1–10 messages**
* Uses **long polling by default** 

📌 Important:

* Visibility timeout ≈ **6× Lambda timeout**

---

### 🗄️ 3. SQS → DynamoDB (Write Buffer)

* Prevents throttling
* Queue absorbs spikes
* Worker writes to DB

---

### 🔁 4. Request–Response Pattern

* Request Queue → Worker → Response Queue
* Fully decoupled system

---

## 🔄 Queue Types

### 🟢 Standard Queue

* Unlimited throughput
* **At least once delivery**
* Possible duplicates
* No ordering guarantee

---

### 🟡 FIFO Queue (First-In-First-Out)

* Ordered processing
* No duplicates
* Limited throughput:

  * 300 msg/sec (no batching)
  * 3000 msg/sec (with batching) 

---

## 💀 Dead Letter Queue (DLQ)

### 🚨 Problem

Message fails repeatedly → infinite retry loop

### ✅ Solution: DLQ

* After **MaxReceiveCount**, message goes to DLQ
* Used for:

  * Debugging
  * Error analysis

📌 Important rules:

* FIFO → FIFO DLQ
* Standard → Standard DLQ 

---

### 🔁 Redrive Feature

* Move messages **back to main queue**
* After fixing issue

---

## ⏱️ Visibility Timeout

* When a consumer reads a message:

  * It becomes **invisible temporarily**
* If not processed → message reappears

---

## ⚠️ Idempotency (VERY IMPORTANT)

Because SQS can deliver messages multiple times:

👉 Your consumer **must be idempotent**

### ❌ Bad:

* Insert duplicate rows in DB

### ✅ Good:

* Use **upsert (same primary key)**
* No duplicate effect

📌 Key idea:
Same message processed twice → same result 

---

## 🔗 Lambda + SQS Special Notes

* Lambda polls SQS (not push)
* Configure:

  * Queue
  * Batch size

📌 DLQ handling:

* Set DLQ on **SQS queue (not Lambda)**
