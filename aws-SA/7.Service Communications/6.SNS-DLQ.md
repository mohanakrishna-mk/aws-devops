Here’s a **deep, exam-focused breakdown of SNS Message Delivery Retries & Dead Letter Queues (DLQ)** 👇

---

## 🔁 SNS Message Delivery Retries

When SNS delivers a message to a subscriber and it **fails**, SNS **automatically retries**.

---

### ⚙️ For AWS-Managed Endpoints

Examples:

* SQS
* Lambda
* Kinesis Data Firehose

👉 SNS uses a **predefined retry policy**

#### 🔄 Retry Phases

1. **Immediate retries**

   * 3 attempts (no delay)

2. **Pre-backoff**

   * 2 retries (1 second apart)

3. **Backoff phase**

   * 10 retries (exponential delay: 1s → 20s)

4. **Post-backoff**

   * ~100,000 retries (every 20 seconds)

📌 Total:
👉 ~100,000+ retries over ~23 days 

👉 That’s **very high durability**

---

## 🌐 For HTTP/HTTPS Endpoints

👉 You can define a **custom delivery policy**

### 🎯 You control:

* Number of retries
* Min / max delay
* Backoff strategy
* Throttling

📌 Important exam point:
👉 **Only HTTP/HTTPS supports custom retry policies**

---

## 💀 SNS Dead Letter Queue (DLQ)

### 🚨 Problem

After all retries → message still fails

👉 Without DLQ = message is **lost**

---

### ✅ Solution: DLQ

* Stores failed messages
* Allows debugging & reprocessing

---

## ⚠️ KEY DIFFERENCE (VERY IMPORTANT)

👉 In SNS:

❌ DLQ is NOT attached to topic
✅ DLQ is attached to **subscription**

📌 From your notes:

* Each subscriber can have its own DLQ 

---

## 🧩 Architecture Example

SNS Topic →

* Email Subscription → DLQ1
* HTTP Subscription → DLQ2

👉 Each failure is handled independently

---

## 📦 DLQ Type

* Must be **SQS queue**
* If SNS FIFO → DLQ must be FIFO

---

## 🧠 Why DLQ is Important

* Prevent message loss
* Debug failed deliveries
* Replay messages later

---

## ⚡ Key Differences vs SQS DLQ

| Feature     | SNS DLQ          | SQS DLQ            |
| ----------- | ---------------- | ------------------ |
| Attached to | Subscription     | Queue              |
| Trigger     | Delivery failure | Processing failure |
| Use case    | Push failure     | Consumer failure   |

---

## 🎯 When to Use (Exam Triggers)

Use SNS DLQ when:

* “Handle failed notifications”
* “Ensure no message loss”
* “Debug delivery issues”
* “HTTP endpoint failures”

---

## 🧠 Simple Mental Model

👉 SNS retries = **“keep trying delivery”**
👉 DLQ = **“store if all retries fail”**

---

## ⚡ One-Line Summary

👉 **SNS retries messages aggressively, and DLQs capture messages that still fail after retries**
