# 🚀 Why Transit Gateway Exists

From your file 

👉 Problem:

* Too many VPCs
* Too many peering connections
* Too many VPNs

👉 Result:

> Network becomes messy and unmanageable

---

# 🧠 Core Idea (Very Important)

### 🕸 Before (Peering)

```id="bdbs1"
A ↔ B
A ↔ C
B ↔ C
```

👉 Gets complex quickly

---

### ⭐ After (Transit Gateway)

```id="3bh2z"
        TGW
      /  |  \
     A   B   C
```

👉 **Hub-and-spoke model**

---

# 🔑 Key Features of Transit Gateway

From your notes 

### ✅ 1. Transitive Routing (BIG DIFFERENCE)

* A → TGW → B → TGW → C
  ✔ Works

👉 Unlike VPC peering

---

### ✅ 2. Centralized Networking

* One place to manage:

  * VPCs
  * VPN
  * Direct Connect

---

### ✅ 3. Works Across:

* Accounts (via RAM)
* Regions (via TGW peering)

---

### ✅ 4. Route Tables at TGW Level

👉 You control:

* Which VPC can talk to which

---

### ✅ 5. Supports IP Multicast

👉 Only TGW supports this in AWS

---

# 🔥 Real Architecture (IMPORTANT)

From your file 

## Central NAT Architecture (This is gold)

```id="yqhrl8"
App VPC 1 \
             → TGW → Egress VPC → NAT → Internet
App VPC 2 /
```

---

### 💡 What’s happening:

* App VPCs are **private**
* No NAT inside them
* All traffic goes:
  → TGW
  → Egress VPC
  → NAT Gateway
  → Internet

---

### ✅ Why this is powerful:

* Centralized internet control
* Cost optimization
* Better security

---

### ⚠️ Important:

👉 This works ONLY with TGW
❌ Not possible with VPC Peering

---

# 🔐 Traffic Control (Hidden Power)

From your notes 

You can create **multiple TGW route tables**:

### Example:

* Prod VPCs → only talk to Prod
* Dev VPCs → only talk to Dev

👉 Full isolation

---

# 🏢 Multi-Account Setup

Using:

* **AWS RAM (Resource Access Manager)**

Flow:

1. Create TGW in Account A
2. Share via RAM
3. Account B attaches its VPC

👉 Clean enterprise architecture

# 🌍 Cross-Region Connectivity

👉 Global architecture

# 🔄 TGW vs VPC Peering (Must Know)

| Feature         | VPC Peering | Transit Gateway |
| --------------- | ----------- | --------------- |
| Transitive      | ❌           | ✅               |
| Central control | ❌           | ✅               |
| Scale           | Limited     | Massive         |
| Cross-account   | ✅           | ✅               |
| NAT sharing     | ❌           | ✅               |
| Complexity      | High        | Low             |

---

# 💡 When to Use What

### Use VPC Peering:

* 2–3 VPCs
* Simple setup

---

### Use Transit Gateway:

* Many VPCs
* Multi-account
* Hybrid (on-prem + cloud)
* Central NAT / firewall

---

# ⚠️ Important Exam + Interview Points

👉 Transit Gateway:

* Supports **edge-to-edge routing**
* Uses **attachments**
* Has its own **route tables**

👉 Peering:

* No transitive routing
* No edge-to-edge

---
