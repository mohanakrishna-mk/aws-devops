Here’s a **clean, exam-focused breakdown of DynamoDB** — this is one of the most important services for **high-scale architectures**.

---

# 🔹 What is DynamoDB?

👉 Amazon DynamoDB

* Fully managed **NoSQL database**
* Serverless (no capacity planning required)
* Massive scale → up to **millions of requests/sec**

📌 Similar concept to Apache Cassandra 

---

# 🔹 Core Features

* No schema (flexible attributes)
* Max item size → **400 KB**
* Supports:

  * CRUD operations
  * Transactions (ACID)
  * Strong / eventual consistency

📌 Larger data → store in S3 + reference in DynamoDB 

---

# 🔹 Capacity Modes

## ⚙️ Provisioned

* Predefine read/write capacity
* Cheaper for predictable workloads

---

## ⚡ On-Demand

* Auto scales instantly
* Pay per request

👉 Best for:

* Unpredictable traffic

---

# 🔥 Data Model (Very Important)

---

## 🧱 Table Structure

* Table → Items (rows) → Attributes (columns)

---

## 🔑 Primary Key Options

---

### 1. Partition Key (Simple)

👉 Must be unique

Example:

```
user_id → unique per user
```

---

### 2. Composite Key

👉 Partition key + Sort key

Example:

```
user_id + timestamp
```

👉 Allows:

* Multiple items per partition
* Sorting

---

# 🔥 Indexing (Exam Favorite)

---

## 🔹 LSI (Local Secondary Index)

* Same partition key
* Different sort key
* Must be created at table creation

---

## 🔹 GSI (Global Secondary Index)

* Different partition key
* Optional sort key
* Can be added later

---

📌 KEY RULE:
👉 You can ONLY query using keys or indexes

❌ No flexible queries like SQL 

---

# 🔹 Advanced Features

---

## ⏳ TTL (Time to Live)

* Auto delete items after time

---

## 🔄 Streams

* Capture changes (insert/update/delete)
* Retention → 24 hours

👉 Common pattern:

* Trigger AWS Lambda

---

## 🌍 Global Tables

* Multi-region replication
* Active-active

👉 Low latency + DR

---

## 📊 Streams + Kinesis

👉 Amazon Kinesis

* Extend retention (up to 365 days)
* Real-time analytics

---

# 🔹 S3 + DynamoDB Pattern (Important)

👉 Problem:

* S3 has no search

👉 Solution:

1. Upload file → S3
2. Trigger Lambda
3. Store metadata in DynamoDB

👉 Enables:

* Search
* Filtering
* Analytics

---

# 🔥 DAX (DynamoDB Accelerator)

👉 Amazon DynamoDB Accelerator (DAX)

* In-memory cache for DynamoDB
* Microsecond latency

---

## 🔄 How it works

* App → DAX → DynamoDB

👉 Transparent caching

---

## 🧠 Benefits

* Fix **hot key problem**
* Reduce throttling
* Faster reads

---

# 🔹 DAX vs ElastiCache (Exam Trick)

---

## 🟢 Use DAX when:

* Caching DynamoDB queries
* No code change required

---

## 🔵 Use ElastiCache when:

* Custom caching logic
* Store computed results
* Complex aggregation

---

# 🔥 Key Comparison

| Feature  | DAX            | ElastiCache         |
| -------- | -------------- | ------------------- |
| Purpose  | DynamoDB cache | General cache       |
| Setup    | Simple         | More control        |
| Use case | Query caching  | Computation caching |

---

# 🧠 Memory Trick

* **DynamoDB = Scale**
* **DAX = Speed**
* **ElastiCache = Flexibility**

---

# ⚡ Exam shortcuts

* “Massive scale DB” → DynamoDB
* “Flexible schema” → DynamoDB
* “Need query-based access” → Index (GSI/LSI)
* “Cache DynamoDB reads” → DAX
* “Cache computed results” → ElastiCache
* “Search S3 data” → DynamoDB + Lambda

---

# 🚀 Real-world architecture

👉 High-scale app:

* Frontend → CloudFront
* Backend → Lambda
* DB → DynamoDB
* Cache → DAX
* Files → S3
