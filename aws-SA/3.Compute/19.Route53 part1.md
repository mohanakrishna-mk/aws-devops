Route 53 looks like “just DNS,” but in AWS exams it’s really about **traffic control strategies**. Let’s simplify it into patterns you can recognize instantly.

---

# 🧠 1) Core Record Types (Foundation)

Amazon Route 53

| Record | Purpose         |
| ------ | --------------- |
| A      | Domain → IPv4   |
| AAAA   | Domain → IPv6   |
| CNAME  | Domain → Domain |
| NS     | Name servers    |

---

## ⚠️ Important Rule

👉 You **cannot use CNAME at root domain**

```text id="0kq6lm"
example.com ❌ CNAME
www.example.com ✅ CNAME
```

---

## ✅ Solution → Alias Record

👉 AWS-specific DNS shortcut

* Works at root domain
* Points to AWS resources (ALB, CloudFront, etc.)
* Free + supports health checks

---

# ⚡ 2) TTL (Time To Live)

👉 Controls DNS caching

---

## Trade-off:

| TTL  | Effect                           |
| ---- | -------------------------------- |
| High | Less DNS cost, slower updates    |
| Low  | Faster updates, more DNS queries |

---

👉 Key idea:

```text id="8gn2mx"
Low TTL = flexibility
High TTL = stability
```

👉 Split traffic by %

Example:

* 70% → v1
* 30% → v2

---

👉 Use cases:

* Canary deployment
* A/B testing

---

## 🔹 Latency-Based Routing

👉 Send user to **lowest latency region**

---

Example:

* India → Mumbai
* US → Virginia

---

👉 Based on **actual latency**, not distance 

---

## 🔹 Failover Routing

👉 Active / Passive setup

```text id="v8h9k3"
Primary (healthy) → used
Primary fails → Secondary used
```

---

👉 Requires health checks

---

## 🔹 Geolocation Routing

👉 Based on **user location (country/region)**

---

Example:

* Germany → German server
* France → French server

---

👉 Used for:

* Compliance
* Localization

---

## 🔹 Geoproximity Routing (Advanced)

👉 Based on:

* Distance + bias

---

## 🔥 Bias Concept

```text id="3g9xjl"
Increase bias → more traffic
Decrease bias → less traffic
```

---

👉 Used with Traffic Flow

---

## 🔹 Multi-Value Routing

👉 Returns multiple healthy IPs

---

## ⚠️ Important

```text id="hx3jq7"
Multi-value ≠ load balancer
```

---

👉 Just filters unhealthy endpoints

---

## 🔹 IP-Based Routing

👉 Based on **client IP ranges (CIDR)**

---

Example:

* ISP A → server 1
* ISP B → server 2

---

👉 Use cases:

* Cost optimization
* ISP routing

---

# 🎯 4) Decision Guide (Exam Gold)

```text id="g6qlhp"
Canary / split traffic → Weighted
Closest region → Latency
Backup system → Failover
Country-based → Geolocation
Fine tuning → Geoproximity
Multiple healthy endpoints → Multi-value
Known IP ranges → IP-based
```
