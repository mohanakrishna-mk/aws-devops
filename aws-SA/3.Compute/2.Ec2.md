# 🧠 1) EC2 Instance Families (Core Concept)

Amazon EC2

Think of instance families as **“what resource is optimized?”**

| Family    | Best For  | Easy Memory           |
| --------- | --------- | --------------------- |
| **R**     | High RAM  | Redis, caching        |
| **C**     | High CPU  | Compute-heavy apps    |
| **M**     | Balanced  | General apps          |
| **I**     | High I/O  | Databases (fast disk) |
| **G**     | GPU       | ML / video            |
| **T2/T3** | Burstable | Cheap + spikes        |

👉 Key exam idea:
Pick **right family = better performance + lower cost**

# 💰 3) EC2 Pricing Models (Classic exam question)

| Type           | Use Case                |
| -------------- | ----------------------- |
| On-Demand      | Short, unpredictable    |
| Spot           | Cheap but interruptible |
| Reserved       | Long-term (1–3 years)   |
| Convertible RI | Change instance type    |
| Dedicated Host | License compliance      |

👉 Key logic:

* Stable workload → **Reserved**
* Fault-tolerant → **Spot**
* Testing → **On-demand**

# 📊 5) EC2 Monitoring (Important gotcha)

Amazon CloudWatch

Default metrics:

* CPU
* Network
* Disk (instance store only)
* Status checks

❗ Important:

* ❌ RAM is NOT included
* ✅ Must send custom metric

# 🛠️ 7) EC2 Auto Recovery

Flow:

```id="p4f7k2"
CloudWatch Alarm → detects failure → triggers recovery
```

What stays SAME after recovery:

* IP address
* Elastic IP
* Metadata
* Placement group 

👉 This is super important for **high availability**

# 🧩 Final Mental Model (Remember This)

When designing EC2:

1. **Pick instance type** → CPU / RAM / I/O
2. **Pick pricing** → cost optimization
3. **Pick placement group** → performance vs availability
4. **Add monitoring** → CloudWatch
5. **Add recovery** → auto-healing
