This is one of the **most confusing but high-yield Route 53 topics**: Hybrid DNS.
Once you see the pattern, it becomes very easy.

---

# 🧠 What is Hybrid DNS?

👉 Connecting DNS resolution between:

* AWS VPC
* On-premises network

---

## 🎯 Goal

```text id="6wq4mv"
AWS ↔ On-prem DNS resolution
```

---

# ⚙️ Default Behavior (Important)

Amazon Route 53

👉 Inside VPC:

* Resolves:

  * EC2 private DNS
  * Private Hosted Zones
  * Public domains

👉 Works automatically

---

❗ Problem:

* Cannot resolve **on-prem DNS**
* On-prem cannot resolve **AWS private DNS**

---

# 🔁 Solution: Resolver Endpoints

👉 Two types:

| Type     | Direction     |
| -------- | ------------- |
| Inbound  | On-prem → AWS |
| Outbound | AWS → On-prem |

---

## 🧠 What it does

```text id="m8pl02"
On-prem DNS → AWS private DNS
```

---

## ⚙️ Flow

1. On-prem DNS receives query
2. Doesn’t know domain
3. Forwards to:

   * Inbound endpoint (ENI IPs)
4. Route 53 resolver answers

---

## 🎯 Example

```text id="gpt2p1"
app.aws.private → resolved from AWS
```

---

## 🔑 Key Idea

```text id="dwr6sh"
Inbound = external → AWS
```

## 🧠 What it does

```text id="n5i3qj"
AWS → On-prem DNS resolution
```

---

## ⚙️ Flow

1. EC2 queries Route 53
2. Rule matches domain
3. Query forwarded to:

   * On-prem DNS

---

## 🎯 Example

```text id="3r2czd"
web.onprem.private → resolved from on-prem
```

---

## 🔑 Key Idea

```text id="ymwzql"
Outbound = AWS → external
```

---

# ⚙️ 3) Resolver Rules (Very Important)

👉 Used with outbound endpoints

---

## 🔹 Conditional Forwarding

```text id="2x0y4v"
If domain = onprem.private → forward to on-prem DNS
```

---

## 🔹 Rule Matching

👉 Most specific rule wins 

---

## 🔹 Types

* Forwarding rules
* System rules
* Auto-defined rules

---

# ⚡ 4) Architecture Requirements

👉 You need network connection:

* VPN
* Direct Connect

---

👉 Endpoints:

* Created in **2 AZs** (high availability)
* Use ENIs

---

👉 Performance:

* ~10,000 queries/sec per IP 

---

# 🔄 5) Full Hybrid Flow

```text id="4t7plk"
On-prem → inbound → AWS DNS
AWS → outbound → on-prem DNS
```

---

# 🧠 Final Mental Model

```text id="a6l2hy"
Inbound → AWS answers
Outbound → AWS asks
```

---

# 🎯 Exam Traps

---

👉 “On-prem needs to resolve AWS private domain”
→ ✅ Inbound endpoint

---

👉 “EC2 needs to resolve on-prem domain”
→ ✅ Outbound endpoint + rule

---

👉 “Hybrid DNS both ways”
→ ✅ Use BOTH endpoints

---

👉 “Forward specific domain only”
→ ✅ Conditional forwarding rule

---

👉 “Multiple rules match”
→ ✅ Most specific wins

---

# 🧩 Easy Memory Trick

```text id="h2pt1p"
Inbound = coming into AWS
Outbound = going out of AWS
```

---

# 🔥 Real-World Insight

In real architectures:

* Inbound → on-prem apps call AWS services
* Outbound → AWS apps call on-prem systems
* Both together → full hybrid
