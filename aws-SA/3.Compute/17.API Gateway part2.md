This is where API Gateway becomes **“production-ready”**—you’re now dealing with **rate limiting, real-time apps, and private access**. These are all common exam traps.

# 🧠 1) Usage Plans + API Keys (Control Access)

Amazon API Gateway

👉 Think of this as:

```text
Who can call your API + how much + how fast
```

---

## 🔹 Usage Plan

Defines:

* Which APIs / stages
* Throttling (requests/sec)
* Quota (total requests/day/month)

---

## 🔹 API Keys

👉 Unique identifier for each client

* Alphanumeric string
* Sent in request headers

---

## 🎯 Why this matters

👉 You can:

* Monetize APIs
* Limit abuse
* Track usage per client

---

## ⚠️ Important

👉 API Keys are **NOT authentication**

* Just identification

---

## 🚨 429 Error (Very Important)

👉 Means:

* Too many requests

Types:

* Per API key
* Account-level throttling

👉 Fix:

* Retry with exponential backoff

---

✅ Internal microservices
✅ Secure backend APIs
✅ No public internet exposure

# 🎯 Exam Traps

---

👉 “Limit API usage per client”
→ ✅ Usage Plan + API Keys

---

👉 “Real-time bidirectional communication”
→ ✅ WebSocket API

---

👉 “API only accessible inside VPC”
→ ✅ Private API + VPC Endpoint

---

👉 “Clients exceed rate limit”
→ ✅ 429 + exponential backoff

---

# 🧠 Final Mental Model

```text
Usage Plan → control traffic
WebSocket → real-time apps
Private API → internal-only access
```
