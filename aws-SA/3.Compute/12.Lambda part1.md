# 🧠 What Lambda *really is*

AWS Lambda

👉 Not just “serverless”
👉 It’s an **event processor with strict limits**

```text
Event → Lambda → Short execution → Exit
```

---

# 🔗 How to Think About Integrations

Instead of memorizing services, group them:

## 🔹 Event Sources

* Amazon S3 → file events
* Amazon DynamoDB → stream changes
* Amazon EventBridge → schedules

## 🔹 Request-Based

* Amazon API Gateway → REST calls

## 🔹 Queue / Async

* Amazon SQS
* Amazon SNS

👉 Lambda = **reaction layer** 

---

# ⚡ The Most Important Rule

```text
Lambda = SHORT + STATELESS + EVENT-DRIVEN
```

Break any of these → wrong choice

---

# ⏱️ Limits (Where Questions Come From)

Instead of memorizing numbers, remember impact:

| Limit          | What it means       |
| -------------- | ------------------- |
| 15 min timeout | Not for long jobs   |
| Memory → CPU   | More RAM = more CPU |
| Concurrency    | Can throttle        |
| Small payload  | Not for large data  |

---

👉 Key takeaway:

```text
If execution time grows → move away from Lambda
```

---

# 🔥 Concurrency (Where people fail)

### Problem:

One function consumes all capacity

### Result:

* Other functions → throttled

---

### Fix:

👉 **Reserved concurrency**

```text
Critical API → reserved capacity
Background jobs → limited capacity
```

👉 Guarantees fairness 

---

# 🚦 Throttling Strategy (Real-world)

Instead of retry chaos:

👉 Put buffer:

```text
Client → SQS → Lambda
```

✔ Smooth traffic
✔ Prevent spikes

# 🧠 When Lambda FAILS (Critical insight)

Your notes say “don’t use for long jobs”—correct, but incomplete.

❌ Also avoid when:

* Need **persistent connections**
* Need **GPU / heavy compute**
* Need **shared memory/state**

---

👉 Use instead:

* ECS / Fargate
* Batch
