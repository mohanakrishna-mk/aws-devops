# 🧠 The Standard AWS Architecture Flow

## 1) Users → DNS

Amazon Route 53
Users never directly hit your servers—they first resolve your domain.

* Converts `yourapp.com` → IP / AWS endpoint
* Can do routing (latency-based, failover, geo routing)

👉 Think: **Entry gate**

## 3) Web Layer (Entry to Backend)

This is where requests go **after CDN (or directly)**:

* Elastic Load Balancing

  * ALB → HTTP apps
  * NLB → high performance / TCP
* Amazon API Gateway → serverless APIs

👉 Think: **Traffic controller**

## 5) Database Layer (State)

* Amazon RDS → SQL

# 🔁 Putting It All Together (Real Flow)

A typical request:

```
User → Route53 → CloudFront → Load Balancer → EC2/Lambda
     → Cache (Redis) → Database (RDS/DynamoDB)
```

---

# ⚡ Key Insight (This is what exam tests)

You are NOT memorizing services.

You are choosing based on:

| Requirement             | Choose               |
| ----------------------- | -------------------- |
| High traffic global app | CloudFront           |
| Serverless API          | API Gateway + Lambda |
| Relational data         | RDS                  |
| Massive scale key-value | DynamoDB             |
| Faster reads            | ElastiCache          |
| Decouple systems        | SQS/SNS              |

---

# 🧩 Real DevOps Thinking (important for you)

Since you're doing pipelines + EC2:

Start with simple architecture:

```
Route53
 → ALB
   → EC2 (Auto Scaling)
     → RDS
     → S3 (assets)
```

Then gradually evolve to:

* Add CloudFront
* Add Redis cache
* Add SQS for async jobs
* Move to containers (ECS/Fargate)
