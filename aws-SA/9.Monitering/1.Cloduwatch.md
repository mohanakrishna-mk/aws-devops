# ☁️ AWS CloudWatch — Deep Dive (Simplified)

## 📊 1. CloudWatch Metrics

These are **automatic measurements from AWS services**.

### Key points:

* Most AWS services (EC2, RDS, etc.) send metrics automatically
* For **Amazon EC2**:

  * Default (standard): **5-minute intervals**
  * Detailed monitoring: **1-minute intervals**

### ⚠️ Important limitation:

* ❌ RAM usage is NOT included
* ✅ You must create a **custom metric** using CloudWatch Agent

### Custom Metrics:

* Default: **1-minute resolution**
* High resolution: **1-second (very important for real-time systems)**

---

## 🚨 2. CloudWatch Alarms (Very Important)

CloudWatch alarms react when a metric crosses a threshold.

### They can trigger ONLY 3 types of actions:

### 1️⃣ EC2 Actions

* Stop
* Terminate
* Reboot
* Recover (⚡ exam favorite)

👉 Example:
If instance status check fails → auto **recover instance**

---

### 2️⃣ Auto Scaling Actions

* Scale IN / OUT based on load
* Used with Auto Scaling Groups

---

### 3️⃣ SNS Notifications

Using **Amazon SNS**:

* Send email alerts
* Trigger Lambda
* Fan-out to multiple services

---

## 🔗 3. EventBridge Integration (Advanced Power)

CloudWatch alarms can send events to:

👉 **Amazon EventBridge**

This unlocks:

* Trigger Lambda
* Start Step Functions
* Send to Kinesis
* Build automation pipelines

👉 Think of it as:

> “CloudWatch Alarm → EventBridge → Anything in AWS”

---

## 📊 4. CloudWatch Dashboards

* Visualize metrics + alarms
* Can show **multi-region data**
* Useful for monitoring entire systems

---

## 🧪 5. CloudWatch Synthetics Canary (Very Important Concept)

This is like a **robot user testing your app**.

### Uses:

* Test APIs
* Monitor websites
* Simulate real user behavior

---

### 🔁 How it works:

* Runs scripts (Node.js / Python)
* Executes every minute (or schedule)
* Uses headless Chrome

---

### Example flow:

1. Canary calls your API regularly
2. If response is slow or wrong → detects issue
3. Triggers CloudWatch Alarm
4. Alarm triggers Lambda
5. Lambda fixes system (e.g., reroute traffic)

---

### 🔗 Example services involved:

* **Amazon Route 53** → routing traffic
* **AWS Lambda** → automation

---

### 🔧 Canary Blueprints:

Prebuilt templates:

* Heartbeat (check if site is up)
* API testing
* Broken link checker
* Visual comparison (UI changes)
* Workflow testing (login flows)

---

## 🧠 Real-World Understanding

### Without Canary:

* You wait for users to complain 😬

### With Canary:

* You detect problems **before users notice** 🚀
