Here’s a **big-picture, exam-focused breakdown of AWS scaling limits & architecture strategy** — this lecture is about **how to handle massive traffic efficiently**.

---

# 🔥 Core Idea

👉 **Handle requests as early as possible (left side of architecture)**
👉 Avoid going deep into backend (expensive + slow)

📌 More right → more:

* Latency
* Cost
* Complexity 

---

# 🔹 1. Entry Layer (Global Scale)

## 🌍 Amazon Route 53

* Handles massive global DNS traffic
* No scaling concern

---

## ⚡ Client-side caching

* Browser / app caching
* Reduces requests early

---

# 🔹 2. CDN Layer (VERY IMPORTANT)

## 🚀 Amazon CloudFront

* ~100,000+ requests/sec
* Edge caching

👉 Stops requests before hitting backend

📌 Huge cost + latency savings 

---

# 🔹 3. API / Load Balancer Layer

## ⚖️ ALB (Application Load Balancer)

* Scales automatically
* No warm-up issues

---

## 🔌 API Gateway

* ~10,000 requests/sec (soft limit)
* Has caching

📌 Can reduce backend load

---

# 🔹 4. Compute Layer

| Option        | Scaling            |
| ------------- | ------------------ |
| EC2 (ASG)     | Slower (boot time) |
| ECS / Fargate | Faster             |
| Lambda        | Very fast          |

---

## ⚡ AWS Lambda

* ~1000 concurrent executions (default, scalable)

👉 Best for burst traffic

---

# 🔹 5. Database Layer (Bottleneck Zone)

## 🧱 Traditional DB

* Amazon RDS
* Amazon Aurora

❌ Harder to scale

---

## ⚡ NoSQL

👉 Amazon DynamoDB

* Auto scaling
* On-demand capacity

👉 Best for high-scale workloads

---

# 🔹 6. Caching Layer (CRITICAL)

## 🚀 Amazon ElastiCache

* Redis → up to 200 nodes
* Memcached → sharding

👉 Reduces DB load drastically

---

## ⚡ DynamoDB DAX

* Cache for DynamoDB

---

# 🔹 7. Storage Layer

## 📦 S3

👉 Amazon S3

* 3,500 writes/sec per prefix
* 5,500 reads/sec per prefix

---

## 🔐 KMS (Hidden Limit)

👉 AWS Key Management Service

* ~10,000 requests/sec

⚠️ Can become bottleneck

---

# 🔹 8. Disk Performance

| Storage        | IOPS     |
| -------------- | -------- |
| EBS gp2        | ~16,000  |
| EBS io1        | ~64,000  |
| Instance Store | Millions |

---

# 🔹 9. File System

## 📂 Amazon EFS

* Scales with usage
* Modes:

  * General
  * Max I/O

---

# 🔹 10. Messaging / Decoupling

## 📬 Amazon SQS

* Virtually unlimited

## 📢 Amazon SNS

* Massive scale

---

## ⚠️ SQS FIFO limits:

* ~300 req/sec (or 3000 with batching) 

---

# 🔹 11. Streaming

## 📊 Amazon Kinesis

* Per shard:

  * 1 MB/s in
  * 2 MB/s out

---

# 🔥 MOST IMPORTANT CONCEPT

## 🧠 “Left = Cheap + Fast”

| Layer  | Strategy |
| ------ | -------- |
| Client | Cache    |
| CDN    | Cache    |
| API    | Cache    |
| DB     | Cache    |

👉 Cache as early as possible

---

# 🔴 Why?

* Less backend load
* Lower cost
* Lower latency

---

# ⚡ Exam shortcuts

* “Handle massive traffic” → **CloudFront + caching**
* “Reduce DB load” → **ElastiCache**
* “High scale DB” → **DynamoDB**
* “Decouple system” → **SQS / SNS**
* “Avoid bottleneck” → **Don’t hit DB directly**

---

# 🧠 Memory Trick

👉 **“Cache early, scale easily”**

---

# 🚀 Real-world architecture

👉 High-scale app:

* Route53 → CloudFront
* CloudFront → API Gateway
* API → Lambda
* Data → DynamoDB
* Cache → Redis

👉 Result:

* Massive scale
* Low latency
* Cost optimized
