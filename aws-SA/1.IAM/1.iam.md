## 🔑 Core IAM Concepts (Quick Refresh)

* **Users** → long-term credentials (access key, password)
* **Roles** → temporary credentials via **STS (Security Token Service)**
* **Groups** → collection of users
* **Policies** → define permissions (JSON)

👉 Example: EC2 instance using a role to access S3 (no hardcoded keys)

---

## 🧩 Types of IAM Policies

1. **AWS Managed**

   * Predefined by AWS (e.g. `AdministratorAccess`)
2. **Customer Managed**

   * You create & reuse across users/roles
3. **Inline**

   * Attached to one user/role only (not reusable)

---

## 📄 IAM Policy Structure

Every policy has:

* **Effect** → Allow / Deny
* **Action** → What action (e.g. `s3:GetObject`)
* **Resource** → On what resource
* **Condition** → Optional restrictions

👉 Important rule:

> ❗ **Explicit DENY always wins over ALLOW**

---

## 🎯 Least Privilege (Very Important)

Give only required permissions.

Example:

* ❌ Bad → `Action: "*", Resource: "*"`
* ✅ Good → Only specific service + resource

Tools:

* **Access Advisor** → shows unused permissions
* **Access Analyzer** → finds public/shared resources

---

## 🔐 IAM Roles vs Resource-Based Policies

This is where many people get confused.

### 1️⃣ IAM Role

* You **assume the role**
* You **lose your original permissions**
* You use **role permissions instead**

👉 Example:

* Account A user assumes role in Account B to access S3

---

### 2️⃣ Resource-Based Policy

* Attached directly to resource (like S3 bucket)
* You **keep your own permissions**

👉 Example:

* S3 bucket allows another account directly

---

### 🔥 Key Difference

| Feature               | IAM Role | Resource Policy |
| --------------------- | -------- | --------------- |
| Permission switch     | Yes      | No              |
| Cross-account access  | Yes      | Yes             |
| Keeps original access | ❌ No     | ✅ Yes           |

## ⚙️ Conditions in Policies (Powerful Feature)

You can restrict access using:

* IP → only specific network
* Time → temporary access
* MFA → require MFA
* Tags → based on resource/user tags

👉 Example:

* Allow access **only from office IP**
* Allow access **only if MFA = true**

---

## 🧠 Policy Variables (Very Useful)

Example:

```json
"Resource": "arn:aws:s3:::my-bucket/${aws:username}/*"
```

👉 Each user gets their own folder automatically

---

## 🚧 Permission Boundaries (Advanced but Important)

* Define **maximum allowed permissions**
* Even if policy allows more → boundary restricts it

👉 Used in real orgs when:

* Devs can create users/roles
* But cannot escalate privileges

## 💡 Real Organization Mapping

What you’ll see in companies:

* CI/CD (GitLab/Jenkins) → uses **IAM Role**
* Cross-account deploy → uses **Assume Role**
* S3 sharing → uses **Bucket Policy**
* Security control → uses **Permission Boundaries + SCP**
