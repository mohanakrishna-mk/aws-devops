# 🔹 1. Migrating FSx Windows: Single-AZ ➝ Multi-AZ

👉 With Amazon FSx (Windows File Server), you **CANNOT directly convert** single-AZ to multi-AZ.

---

## ✅ Option 1 — No downtime (Recommended)

* Create new **Multi-AZ FSx**
* Use AWS DataSync to sync data
* Cut over when ready

### ✔ Pros:

* No downtime
* Continuous sync

### ❌ Cons:

* Slower migration

📌 Best when **availability is critical** 

---

## ⚡ Option 2 — Faster (with downtime)

* Take backup of old FSx
* Restore into Multi-AZ
* Switch application

### ✔ Pros:

* Faster

### ❌ Cons:

* Downtime

📌 Best when **speed > availability** 

---

# 🔹 2. Reducing FSx Storage Size (Important Trap)

👉 You **cannot decrease FSx size directly**

---

## ✅ Correct approach

1. Create **new smaller FSx**
2. Use DataSync to migrate data
3. Switch application

📌 Backup/restore WON’T help → size can only increase 

---

## 🧠 Memory:

👉 “FSx → scale up only, never down”

---

# 🔹 3. FSx for Lustre + S3 (Lazy Loading)

👉 Key feature: **Data Lazy Loading**

---

## 🔄 How it works

* Huge dataset stored in Amazon S3
* FSx for Lustre is linked to it

👉 Data is:

* ❌ NOT fully downloaded upfront
* ✅ Loaded **only when requested**

---

## 🚀 Benefits

* Start processing immediately
* Lower cost
* Reduced latency

📌 Perfect for:

* ML workloads
* Big data processing
* Petabyte-scale datasets 

---

## 🧠 Simple Example

* 1 PB dataset in S3
* Job needs only 10 GB

👉 Only 10 GB is loaded into FSx → processed

---

## 🔁 Smart behavior

* Already loaded data → reused
* No re-download unless changed

---

# 🔥 Real-world architecture

👉 ML pipeline:

* Raw data → S3
* Compute → EC2 / HPC cluster
* File system → FSx for Lustre

👉 Flow:

* FSx mounts S3
* Loads only required data
* Writes output back to S3

---

* “Huge dataset + fast processing” → **FSx Lustre + S3 lazy loading**
