# 🔹 1. Amazon EBS (Elastic Block Store)

👉 Think of EBS as a **network-attached hard disk** for EC2.

## Key characteristics

* Attached to **one EC2 instance at a time** (except multi-attach)
* Bound to a **single Availability Zone**
* **Persistent storage** → data survives instance stop/start
* Can **resize (only increase size)** without detaching
* Supports **snapshots (backups)**

📌 Important:

* Snapshots are **incremental**
* Stored in S3 (but not directly accessible)
* Used to **move volumes across AZ/regions** 

---

## 🔸 EBS Volume Types

| Type      | Use case                                    |
| --------- | ------------------------------------------- |
| gp2 / gp3 | General purpose (most common)               |
| io1 / io2 | High performance, low latency               |
| st1       | Throughput-heavy workloads (big data, logs) |
| sc1       | Cheap cold storage                          |

📌 Boot volumes → only **gp2, gp3, io1, io2** 

---

## 🔸 Advanced Features

### ✅ Snapshots

* Backup EBS volumes
* Can copy across **regions (DR strategy)**
* Used to create **AMIs**

### ✅ Fast Snapshot Restore (FSR)

* Avoid slow reads after restore
* Pre-warms volume automatically

### ✅ Encryption

* Account-level setting (per region)
* Encrypts:

  * volumes
  * snapshots

### ✅ Data Lifecycle Manager (DLM)

* Automates:

  * snapshot creation
  * retention
  * deletion

📌 Difference:

* **DLM → EBS only**
* **AWS Backup → multiple services** 

---

## 🔸 EBS Multi-Attach (special case)

* Only for **io1 / io2**
* Same volume → multiple EC2 instances
* Same AZ only
* Requires **cluster-aware filesystem**

👉 Used for:

* High availability apps
* Concurrent write systems

---

# 🔹 2. EC2 Instance Store

👉 Think of this as a **temporary local disk physically attached to the server**

## Key characteristics

* **Very high performance (millions of IOPS)**
* **Ephemeral storage (temporary)**
* Data lost when:

  * instance stops
  * instance terminates

📌 Data survives only **reboot**, not stop/terminate 

---

## 🔸 Pros

* Ultra-fast (better than EBS)
* Good for:

  * cache
  * buffer
  * temp data
  * scratch processing

## 🔸 Cons

* No persistence
* Cannot resize
* No snapshot capability
* Must manage backups manually

---

# 🔥 3. EBS vs Instance Store (Exam Gold Table)

| Feature     | EBS             | Instance Store |
| ----------- | --------------- | -------------- |
| Type        | Network storage | Physical disk  |
| Persistence | ✅ Yes           | ❌ No           |
| Performance | High            | Very High      |
| Resize      | ✅ Yes           | ❌ No           |
| Backup      | Snapshots       | Manual         |
| Use case    | DB, OS, apps    | Cache, temp    |

---

# 🧠 Real-world decision shortcut

* Use **EBS** → anything important (DB, app, OS)
* Use **Instance Store** → speed-critical temporary data

---

👉 **EBS = Durable**
👉 **Instance Store = Disposable**
