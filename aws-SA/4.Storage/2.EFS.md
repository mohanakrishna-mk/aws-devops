Here’s a **clear, practical breakdown of Amazon EFS (Elastic File System)** from your notes, simplified for **exam + real-world use**.

---

# 🔹 1. What is EFS?

👉 **EFS = Managed Network File System (NFS)**

* Can be mounted on **multiple EC2 instances at the same time**
* Works across **multiple Availability Zones**
* Fully **managed, scalable, highly available**

📌 Key idea:
👉 **EBS = single instance block storage**
👉 **EFS = shared file system for many instances**

---

# 🔹 2. Core Features

## ✅ Multi-instance access

* Many EC2 instances → same file system
* Even across AZs

## ✅ Fully managed + scalable

* Auto scales from **GB → Petabytes**
* No capacity planning needed

## ✅ Pay per usage

* Pay only for **used storage**
* Unlike EBS (pay for provisioned size) 

## ✅ Works with on-prem

* Via:

  * VPN
  * Direct Connect

---

# 🔴 Important Limitation (EXAM TRAP)

❌ Works only with **Linux (POSIX)**
❌ Not supported for Windows 

---

# 🔹 3. Use Cases

* Content management systems (CMS)
* Shared storage between services
* Web servers (like WordPress)
* Data sharing across apps

---

# 🔹 4. Performance Modes

| Mode            | Use case                  |
| --------------- | ------------------------- |
| General Purpose | Low latency (default)     |
| Max I/O         | High throughput, big data |

---

# 🔹 5. Throughput Modes

| Mode        | Description                |
| ----------- | -------------------------- |
| Bursting    | Throughput grows with size |
| Provisioned | Fixed throughput           |
| Elastic     | Auto scales with usage     |

👉 Use **Elastic** for unpredictable workloads

---

# 🔹 6. Storage Classes (Cost Optimization)

| Tier                   | Use                    |
| ---------------------- | ---------------------- |
| Standard               | Frequently accessed    |
| IA (Infrequent Access) | Less used data         |
| Archive                | Rare access (cheapest) |

📌 Lifecycle policies automatically move data between tiers
👉 Can save up to **90% cost** 

---

# 🔹 7. Availability Options

| Option   | Description                  |
| -------- | ---------------------------- |
| Standard | Multi-AZ (production)        |
| One Zone | Single AZ (cheaper, dev use) |

---

# 🔹 8. Security & Access

## ✅ Security Groups

* Control network access

## ✅ IAM + Access Points

* Restrict directory-level access

👉 Example:

* `/config` → App team
* `/data` → Analytics team

## ✅ Encryption

* At rest (KMS)
* In transit

---

# 🔹 9. Mounting EFS

* EC2 → via DNS
* On-prem → must use **ENI IP address** 

---

# 🔹 10. Cross-Region Replication

* Replicates EFS → another region
* Used for:

  * Disaster Recovery
  * Compliance

📌 RPO/RTO = minutes 

---

# 🔥 11. EFS vs EBS vs Instance Store (Ultimate Comparison)

| Feature        | EFS            | EBS                | Instance Store |
| -------------- | -------------- | ------------------ | -------------- |
| Type           | File           | Block              | Block          |
| Multi-instance | ✅ Yes          | ❌ (except special) | ❌              |
| AZ             | Multi-AZ       | Single AZ          | Single host    |
| Persistence    | ✅ Yes          | ✅ Yes              | ❌ No           |
| Scaling        | Auto           | Manual             | Fixed          |
| OS             | Linux only     | Any                | Any            |
| Use case       | Shared storage | DB / OS            | Cache/temp     |

---

# 🧠 Simple Memory Trick

* **EBS → One server disk**
* **EFS → Shared network drive**
* **Instance Store → Temporary ultra-fast disk**

---

# ⚡ Real-world architecture tip

👉 If you're building:

* **React + Node + multiple EC2 servers**
  → use **EFS** for shared uploads

👉 If single server DB:
→ use **EBS**

👉 If caching / temp processing:
→ use **Instance Store**
